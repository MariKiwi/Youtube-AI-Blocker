# Privacy Policy

Last updated: 2026-03-17

This document describes the privacy model for the YouTube AI Blocker project.

This repository hosts the browser extension source, the public website, and the backend API.

## Scope

YouTube AI Blocker is an open source project with three main parts:

- a browser extension
- a backend API and database
- a public website

Because the project is open source, privacy responsibilities depend on who operates a given deployment.

## Open Source and Self-Hosting

If you run your own instance of the API, website, or extension defaults, you are the controller or operator for that deployment and you are responsible for your own privacy disclosures, legal basis, retention rules, and contact details.

If an official public deployment is provided by the project maintainers, this file is the project-level privacy reference for that deployment unless a more specific policy replaces it.

## Data Stored in the Extension

The extension stores some data locally in the browser:

- extension settings such as API server URL, blocking preference, and debug preference
- one anonymous device ID used to identify the browser installation for voting behavior

This data stays in the browser storage area unless the user resets the extension settings, clears browser extension storage, or uninstalls the extension.

The extension does not require account registration.

## Requests Sent by the Extension

When the extension is active on YouTube pages, it can send requests to the configured API server.

### Video lookup requests

The extension may send:

- the YouTube video ID being viewed
- batches of visible YouTube video IDs on supported feed pages
- the local anonymous device ID with lookup requests when the API response needs to include the current device vote

These requests are used to return the current moderation state for videos, such as `unknown`, `low`, `medium`, `high`, `disputed`, or `unflagged`.

The device ID is included in lookup requests so the API can tell the extension whether this specific browser installation has already voted on a video. That allows the extension to:

- show a visual indicator that you already voted on that video
- render the correct active vote state in the UI
- keep the displayed state consistent after you reload the page or revisit the video

### Flag and vote requests

When a user flags a video as AI-made or votes on an existing entry, the extension sends:

- the YouTube video ID
- the local anonymous device ID
- the action taken, such as `flag`, `up`, or `down`

This data is used to create and update the shared moderation record and to help enforce one vote per device.

The anonymous device ID is only stored as a moderation identifier for vote and flag records. Its purpose is to prevent multiple votes from the same device on the same video and to support the UI state that shows whether the current device has already voted.

## Data Processed by the API

For the official deployment, the API may process:

- YouTube video IDs submitted by the extension
- anonymous device IDs sent by the extension
- flag and vote actions
- moderation timestamps such as record creation and update times
- request and error logs generated by the application runtime
- IP-based rate-limiting data for write actions such as flagging and voting
- normal connection metadata that may be available to the application, host, reverse proxy, CDN, or platform infrastructure

The project currently does not claim that lookup requests are log-free.

### Why lookup requests include device IDs

Lookup requests can include the anonymous device ID because the extension needs to know whether the current browser installation has already voted on a video. Without that identifier, the extension could not reliably show the current device vote state during normal browsing.

### Why moderation timestamps are stored

Moderation records currently include creation and update timestamps. These timestamps help maintain the shared database, support operational inspection of moderation state changes, and provide a basic record of when entries and votes were created or updated.

### Why request and error logging exist

The current server runtime enables request and error logging. This supports debugging, availability monitoring, incident response, and investigation of failures in the API or deployment environment.

### Why IP-based rate limiting exists

The API uses IP-based rate limiting on write endpoints such as `flag` and `vote` to reduce abuse and repeated automated submissions. This is intended to protect the integrity of the shared moderation system.

The current open source codebase is designed around anonymous device-based voting and does not require a named user account.

## Website Data and Cookies

The public website may process standard web request data needed to deliver pages.

If website analytics are configured, the website is designed to keep analytics off until the visitor explicitly opts in through the cookie consent flow.

The website currently uses a consent cookie named `yaib_analytics_consent` to remember whether analytics were accepted or declined.

If analytics are disabled or not configured, no optional analytics script is loaded.

The public website also loads Google Fonts from `fonts.googleapis.com` and `fonts.gstatic.com`. That means Google may receive standard web request information, such as IP address and browser metadata, when the website is opened.

## Legal Basis and Compliance

For official deployments intended for users in the European Union, processing is generally intended to rely on one or more of the following legal bases, depending on the deployment and use case:

- legitimate interests for necessary service operation, abuse prevention, and security
- consent for optional analytics or similar non-essential website processing
- compliance obligations where consent choices or legal requests must be documented

For GDPR / DSGVO compliance, the actual controller for a deployment must ensure that the legal basis, retention periods, international transfers, processor relationships, and user-rights handling are documented correctly for that deployment.

## Sharing and Third Parties

For the official deployment, the project does not sell or intentionally share extension or API moderation data with unrelated third-party data brokers or advertisers.

However, the service can still rely on third parties that are technically involved in operating or publishing the project, such as:

- hosting providers
- reverse proxies or CDNs, if used
- browser stores
- GitHub or release hosting services
- Google Fonts for website font delivery
- Google Search Console verification for search indexing of the website

Those third parties may receive standard technical request data as part of delivering their service.

## Data Retention

Retention depends on the deployment operator.

At a minimum:

- extension local storage remains until the user clears it or removes the extension
- moderation records may remain as long as the operator keeps the shared database active
- server logs, proxy logs, and analytics retention depend on the host and deployment configuration

If you run an official public deployment, publish more specific retention periods once they are defined operationally.
